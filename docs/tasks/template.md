`status` は `todo`（未着手）、`doing`（作業中）、`done`（完了）のいずれかを使用し、それ以外は使わない。

# タスク名

<!-- タスクの内容が端的に分かる名前を書く。例: 記事一覧の表示を整える -->

## Goal

このタスクで達成することを書く。完了後の状態を短く具体的に示す。

## Background

必要に応じて、背景、関連する既存仕様、今回このタスクを行う理由を書く。

## Requirements

- 実装要件を書く。
- 必要なら、画面・データ・動作などの条件を分けて書く。

## Out of scope

- 今回は対応しないことを書く。
- 将来の拡張や別タスクに切り出す内容もここに書く。

## Acceptance criteria

- [ ] 完了条件を書く。利用者の視点で確認できる状態にする。
- [ ] 必要なら、レスポンシブ対応やアクセシビリティなどの条件を追加する。

## Constraints

- 実装上の制約を書く。例: 既存コンポーネントを再利用し、依存パッケージを追加しない。

## Related files

- 関連する可能性があるファイルやドキュメントを書く。
- 実装前に確認してほしい既存ファイルがあれば挙げる。

## Verification

実行するコマンドや、手動で確認する内容を書く。例: `npm run build`、主要画面をブラウザで確認する。
